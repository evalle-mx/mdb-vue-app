Auto-scaling looks at the average utilization of CPU and memory within the evaluation time window. Auto-scaling is triggered based on the average workload of the previous hour. For cluster-tier auto-scaling, it monitors CPU and memory usage.

A cluster scales up if the average CPU or memory utilization exceeds 75% for an hour. It scales down if both CPU and memory utilization are below 50% for 24 hours. Storage auto-scaling increases storage when disk usage reaches 90%, with specific behaviors depending on the cloud provider.

If there are concerns about CPU or memory impact, we recommend manually scaling up the cluster if you anticipate a period of high workload to avoid performance issues.

Consider that escalation is automatically launched via an algorithm based on CPU and Memory usage; however, this is not triggered immediately and is based on statistics for usage.

A cluster must meet two conditions to scale up:

  1. The average CPU utilization or average memory utilization exceeds 75% for 1 hour, and
  2. The cluster has not resumed from a paused state or scaled in the past hour.

That means that the cluster will be modified temporarily to the next tier if it is required or just for the resources that it could need, doing this one node at a time, and no downtime is expected (this may vary if the workload exceeds the current capacity in a peak).

The calculation for CPU utilization is:

 % of normalized user CPU + % normalized kernel CPU

The calculation for Memory utilization is

 (total system memory - free system memory - file system cache - buffer memory) / total system memory

As the System Memory chart in Atlas does not provide free system memory, file system cache, and buffer memory, you can approximate memory utilization with the following formula:

 (total memory - memory available)/total memory * 100 = memory utilization

You can see more details on the Configure Auto-Scaling documentation.

Note that the Atlas auto-scaling feature is designed to address long-term sustained growth rather than sudden known surges in workload. MongoDB is working on improvements to ensure clusters trigger auto-scaling up/down on behalf of our customers. You can configure some characteristics, such as the maximum cluster size and minimum cluster size values, that your cluster can automatically scale to; nevertheless, this feature was not intended for handling very spiky/sudden workloads.

Work-around

Unfortunately, this configuration is not mutable, and it works the same in all the instances; however, a workaround for this is you could Configure an Alert when your CPU usage has additional conditions that notify the admin or someone that could run a script or manually request a tier increment.

Your developer or DBA admin may create an script to run on your end that could take advantage of the MongoDB Atlas Admin API when this receives any specific notification(s).
